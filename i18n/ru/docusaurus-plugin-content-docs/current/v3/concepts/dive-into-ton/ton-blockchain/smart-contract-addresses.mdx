# Адреса смарт-контрактов

Этот документ даёт общее представление об адресах смарт-контрактов в блокчейне TON. Каждый [актор](/v3/concepts/dive-into-ton/ton-blockchain/blockchain-of-blockchains#single-actor), включая кошельки и смарт-контракты, представлен адресом, который используется для всех взаимодействий в блокчейне.

## Компоненты адреса

Каждый адрес в TON состоит из двух основных компонентов:

- **Workchain ID**: 32-битное целое число со знаком, которое обозначает, к какому воркчейну относится контракт<br />
  У мастерчейна значение `-1`, а у бейсчейна `0`.
- 256-битный уникальный идентификатор контракта в его воркчейне. <br />
  Его получают с помощью хеширования начального кода и состояния контракта.

Подробные технические спецификации см. в [документации по адресам смарт-контрактов](/v3/documentation/smart-contracts/addresses/address).

## Форматы адресов

Адреса TON бывают представлены в двух стандартных форматах:

- **«Сырой» формат (raw format)**: прямое представление `workchain_id` и `account_id` (например, `-1:fcb...260`). Предназначен для использования на системном уровне, не включает пользовательские функции безопасности.
- **«Пользовательский» формат (user-friendly format)**: версия в кодировке base64, включающая флаги и контрольную сумму. Этот формат, разработанный для пользовательских интерфейсов, повышает безопасность и предотвращает ошибки.

Эти варианты не взаимозаменяемы. Второй включает важные метаданные, такие как *флаги bounceable*, от которых зависит обработка сообщений неинициализированными аккаунтами. Если сообщение отправлено на неактивный адрес в формате *bounceable*, оно возвращается отправителю. А если *non-bounceable*, его средства будут зачислены аккаунту-адресату.

Для подробного разборка структур адресов, кодирования и флагов обратитесь к [документации по форматам адресов](/v3/documentation/smart-contracts/addresses/address-formats).

## Состояния адресов

Каждый адрес в блокчейне TON существует в одном из четырёх состояний, отражающих его жизненный цикл и рабочий статус:

- `nonexist`: исходное состояние всех потенциальных адресов. У адреса в этом состоянии нет своего кода, данных или баланса.
- `uninit`: адрес получил средства, теперь у него есть баланс, но он не содержит кода или данных смарт-контракта. Он не может выполнять логику.
- `active`: у адреса есть баланс, код и данные смарт-контракта. Он полностью работоспособен и может обрабатывать сообщения.
- `frozen`: плата за хранение контракта превысила его баланс, заморозив его состояние. Адрес не может исполнять транзакции, пока он не будет разморожен, для чего требуется пополнить его баланс и предоставить его состояние.

Состояние адреса определяет его возможности и является основополагающим для обработки транзакций. Дополнительные сведения доступны в [документации по состояниям адресов](/v3/documentation/smart-contracts/addresses/address-states).
